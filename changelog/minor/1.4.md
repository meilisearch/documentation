# v1.4

First released: 2023-09-25

## New Features

### Customize Text Separators
You can now configure word segmentation in Meilisearch using the `nonSeparatorTokens` and `separatorTokens` index settings. This allows you to define which characters should be treated as word separators or non-separators, improving search accuracy for your dataset.

To add characters as word separators:

```bash
curl \
 -X PUT 'http://localhost:7700/indexes/articles/settings/separator-tokens' \
 -H 'Content-Type: application/json' \
 --data-binary '["&sect;", "&sep"]'
```

To specify characters that should not be treated as separators:

```bash
curl \
 -X PUT 'http://localhost:7700/indexes/articles/settings/non-separator-tokens' \
 -H 'Content-Type: application/json' \
 --data-binary '["@", "#", "&"]'
```

### Load User-Defined Dictionaries
You can now enhance word segmentation by using the `dictionary` index setting to include domain-specific terms. This helps improve search accuracy for specialized datasets.

To add custom terms to the dictionary:

```bash
curl \
 -X PUT 'http://localhost:7700/indexes/articles/settings/dictionary' \
 -H 'Content-Type: application/json' \
 --data-binary '["J. R. R.", "J.R.R."]'
```

You can also combine this with `stopWords` and `synonyms` settings:

```json
{
  "dictionary": [
    "J. R. R.",
    "J.R.R."
  ],
  "synonyms": {
    "J.R.R.": [
      "jrr",
      "J. R. R."
    ],
    "J. R. R.": [
      "jrr",
      "J.R.R."
    ],
    "jrr": [
      "J.R.R.",
      "J. R. R."
    ]
  }
}
```

## Improvements

### Enhanced Error Messages for Hidden Fields
Meilisearch now provides clearer error messages when attempting to sort by non-sortable fields. Hidden fields will not be displayed, and you will see a message indicating the available sortable attributes:

```
Available sortable attributes are: price, stock, <..hidden-attributes>.
```

## Other

### Breaking Change for `filter` Parameter
A significant change has been made regarding the use of `\` characters in the `filter` search parameter. Users must now escape all `\` characters in their filter expressions.

For example, to filter on a document with a path containing backslashes, you must now use:

```text
path = "my\\\\test\\\\path\\\\"
```

This change allows for more accurate filtering but requires users to adjust their existing filter expressions if they previously used backslashes. If you need assistance with this change, please reach out for support.
