<Update label="v1.35" description="2026-02-02">

## New Features

### Performance Details in Search Responses
You can now gain insights into the performance of your search queries. By adding the `showPerformanceDetails` parameter to your search requests, you will receive detailed performance metrics in the response.

#### Search Request Example
```http
POST /indexes/<index_uid>/search
```
```json
{
  "q": "glass",
  "showPerformanceDetails": true
}
```

#### Search Response Example
```json
{
 "hits": [
 // hits ..
 ],
 "query": "glass",
 "processingTimeMs": 5,
 "limit": 20,
 "offset": 0,
 "estimatedTotalHits": 1,
 "requestUid": "<uuid>",
 "performanceDetails": {
 "wait for permit": "295.29µs",
 "search > tokenize": "436.67µs",
 "search > resolve universe": "649.00µs",
 "search > keyword search": "515.71µs",
 "search > format": "288.54µs",
 "search": "3.56ms"
 }
}
```

### Performance Details in Multi-Search
The `showPerformanceDetails` parameter is also available in the `POST /multi-search` endpoint, allowing you to analyze the performance of multiple search queries at once.

#### Multi-Search Request Example
```http
POST /multi-search
```
```json
{
  "queries": [
    {
      "indexUid": "<index_uid>",
      "q": "glass",
      "showPerformanceDetails": true
    }
  ]
}
```

#### Multi-Search Response Example
```json
{
 "results": [
 {
 "indexUid": "<index_uid>",
 "hits": [
 // hits ..
 ],
 "query": "glass",
 "processingTimeMs": 5,
 "limit": 20,
 "offset": 0,
 "estimatedTotalHits": 1,
 "requestUid": "<uuid>",
 "performanceDetails": {
 "wait for permit": "295.29µs",
 "search > tokenize": "436.67µs",
 "search > resolve universe": "649.00µs",
 "search > keyword search": "515.71µs",
 "search > format": "288.54µs",
 "search": "3.56ms"
 }
 }
 ]
}
```

### Performance Details in Similar Queries
You can also include the `showPerformanceDetails` parameter in requests to the `POST /indexes/<index_uid>/similar` and `GET /indexes/<index_uid>/similar` endpoints.

#### Similar Request Example


## Improvements

### Enhanced Indexing Performance
The multithreaded post-processing of facets and prefixes is now always enabled, resulting in faster indexing on multi-core machines. This improvement benefits cloud users upgrading to v1.35.0.

## Other

### Breaking Change: Updated Response for Fields Route
The `POST /indexes/<index_uid>/fields` route now returns a structured object instead of an array. The new response includes `results`, `offset`, `limit`, and `total` fields, providing better clarity on the number of fields matching a given filter.

#### Example: Before and After
**Before:**
```json
[
 // fields ..
]
```

**After:**
```json
{
 "results": [
 // [fields .. ]
 ],
 "offset": 0,
 "limit": 20,
 "total": 0
}
```

</Update>

<Update label="v1.34" description="2026-01-26">

## New Features

### Increased Auth Store Database Size
The auth store database size has been increased to 2 GiB, allowing you to index more keys efficiently.

### `useNetwork` Field in Search Requests
You can now use the `useNetwork` field in the `POST /indexes/{:indexUid}/search` request to perform searches as if they were remote federated searches across multiple Meilisearch instances.

Example request:

```json
{
  "q": "Batman dark knight returns 1",
  "filter": "genres IN [Action, Adventure]",
  "facets": [
    "genres"
  ],
  "useNetwork": true,
  "limit": 5
}
```

### `useNetwork` Query Parameter
The `useNetwork` parameter can also be passed in the `GET /indexes/{:indexUid}/search` request, providing the same functionality as in the `POST` request.

### Federated Search with `useNetwork`
You can include `useNetwork` in individual queries within a `POST /multi-search` request, allowing for federated searches across your network.

Example multi-search request:

```json
{
  "federation": {
    "limit": 5
  },
  "queries": [
    {
      "q": "Batman returns",
      "indexUid": "mieli",
      "useNetwork": true
    },
    {
      "q": "Superman returns",
      "indexUid": "mieli",
      "useNetwork": true
    }
  ]
}
```

## Improvements

### Enhanced Federated Search Pagination
You can now use `federation.page` and `federation.hitsPerPage` in federated search requests to enable exhaustive pagination.

### More Efficient Settings Changes
The settings indexer has been optimized for better performance when removing searchable attributes from searchable fields.

### Security Improvements
A low-severity timing attack vulnerability on key comparison has been addressed by implementing constant-time comparison.

## Other

### Breaking Change: Security Restrictions on Non-Global IPs
A breaking change has been introduced to enhance security by forbidding requests to non-global IP addresses. You can allow specific non-global IP networks using the `--experimental-allowed-ip-networks` parameter or the `MEILI_EXPERIMENTAL_ALLOWED_IP_NETWORKS` environment variable. The default behavior is to reject all such requests.

</Update>

<Update label="v1.33" description="2026-01-19">

## New Features

### New `/fields` Endpoint
A new POST `/indexes/{indexUid}/fields` endpoint has been added to retrieve detailed metadata about all fields in an index. This includes information on field configuration such as display, search, filtering, and localization settings.

## Improvements

### Enhanced Vector Store Performance
The vector store has been updated to trigger linear scanning even on larger databases. This improvement leads to better performance and result quality, particularly when the number of filtered candidates is small compared to the total number of documents in the index.

### Parallel Cleanup of Old Field IDs
The process for performing dumpless upgrades for instances prior to v1.32.0 has been optimized through multi-threading database fetches. This enhancement significantly reduces the time required for upgrades, from approximately 2 hours and 50 minutes to under 7 minutes.

### Improved Ranking Scores
A fix has been implemented to ensure that all non-blocking buckets are considered for non-final ranking rules. This change enhances the quality of search results, especially when the search cutoff is triggered and vector search is involved.

## Other

### Security Fix for Dump Import
A security vulnerability related to path traversal in the dump import functionality has been addressed. Open-source users who allow importing dumps from untrusted sources are advised to update to v1.33.1 to mitigate this risk. Cloud users do not need to take any action as there is no evidence of exploitation on Meilisearch Cloud.

</Update>

<Update label="v1.32" description="2026-01-12">

## Improvements

### Skip Field-ID-Based Database Cleanup
You can now opt out of the field ID-based database cleanup when upgrading from versions prior to 1.32.0 by setting the `MEILI_EXPERIMENTAL_DISABLE_FID_BASED_DATABASES_CLEANUP` environment variable.

### Enhanced Search Performance Logging
Comprehensive progress tracking and logging for search operations have been introduced. This feature provides detailed timing information for each step of the search process, improving observability and performance analysis.

### Parallel Document Operations
Document indexing has been accelerated by processing a large number of tasks in batches or records in parallel. This change has resulted in a 7x speedup for inserting four million documents using four CPUs. Note that the `indexedDocuments` field in tasks using `skipCreation` may not accurately reflect the number of document operations performed, but the documents are still correctly indexed.

## Other

### Dumpless Upgrade Handling
The system will no longer panic when performing a dumpless upgrade on empty indexes that have configured embeddings. This change improves stability during upgrades.

### Vector Sort Improvement
The vector sort functionality has been enhanced to ensure that documents with identical similarity scores are grouped together, allowing for correct application of subsequent ranking rules.

### Document Deletion Fix
A bug has been fixed where changing `searchableAttributes` from `["*"]` to a subset of fields left orphaned data in field ID-based databases, which could cause corruption and warnings during search operations.

### Graph Link Rebuilding
The upgrade process now includes a method to rebuild graph links, addressing previously malformed graphs and ensuring proper association of related operations during upgrades.

</Update>

<Update label="v1.31" description="2025-12-22">

## Improvements

### S3-Streaming Snapshots for Enterprise Edition
The S3-streaming snapshots feature is now exclusively available for the Enterprise Edition. A license is required to use this feature in self-hosted solutions. The Community Edition will no longer support S3-streaming snapshots, but on-disk snapshots remain accessible.

### AWS IRSA Authentication for Snapshots
Support for AWS IRSA (IAM Roles for Service Accounts) has been added, allowing the use of short-lived access and secret keys for uploading snapshots to S3. This feature is available under the Enterprise Edition and can be accessed through two experimental CLI parameters.

### Strict Document Update Option
A new optional `skipCreation` boolean query parameter has been introduced for `POST` and `PUT` requests on `/indexes/{index}/documents`. When set to `true`, this parameter allows you to skip the creation of documents that do not exist in the index, preserving existing behavior when set to `false`.

```http
POST /indexes/{index}/documents?skipCreation=true
```

## Other

### Breaking Change: S3-Streaming Snapshots
The S3-streaming snapshot feature is no longer available in the Community Edition, marking a significant change in how snapshots can be managed. Users of the Community Edition who were using this feature between version 1.25 and this release will need to transition to the Enterprise Edition to continue using it.

</Update>

<Update label="v1.30" description="2025-12-15">

## New Features

### Network Sharding Modifications
Meilisearch now allows you to modify the number of participants in sharding without needing to restart the document sending process. This is particularly useful for users of the experimental `network` feature who have enabled automatic sharding.

To set up the initial network, you can send a `PATCH` request to the `/network` endpoint:

```http
PATCH /network
```

With the following JSON body:

```json
{
  "self": "ms0",
  "leader": "ms0",
  "remotes": {
    "ms0": {
      "url": "URL_OF_MS0",
      "searchApiKey": "SEARCH_API_KEY_OF_MS0",
      "writeApiKey": "WRITE_API_KEY_OF_MS0"
    },
    "ms1": {
      "url": "URL_OF_MS1",
      "searchApiKey": "SEARCH_API_KEY_OF_MS1",
      "writeApiKey": "WRITE_API_KEY_OF_MS1"
    }
  }
}
```

### New NetworkTopologyChange Tasks
New tasks have been introduced to manage network topology changes. These tasks will handle the rebalancing of documents when adding or removing remotes from the network.

## Improvements

### Enhanced Error Logging
A new warning will be logged when the `FieldidsWeightsMap` is missing an entry, helping to prevent errors during search requests. This improvement enhances the overall user experience by providing clearer feedback.

## Other

### Breaking Changes in Network Feature
Several breaking changes have been introduced for users of the experimental `network` feature:

1. The `Network` object returned or edited by the `/network` route has been modified:
 - The `sharding` boolean has been removed.
 - A `leader` field has been added as an optional string.
 - A `version` field has been added as a UUID.

2. When a `network.leader` is set, certain routes will fail with a `not_a_leader` error if the target's `network.self` does not match the `network.leader`.

3. The `PATCH /network` route no longer returns a `Network` object but instead spawns a new `NetworkTopologyChange` task.

4. Tasks are now duplicated by the leader to the entire network for specific routes, ensuring consistency across all instances.

5. When importing dumps, the `self` and `leader` fields are dropped from the network.

These changes are crucial for users utilizing the network feature, as they affect how tasks and network configurations are managed.

</Update>

1. Document template rendering failures
2. Embedding request failures (including missing vectors in `userProvided` embedders)

### Metadata Header for Search Requests
You can now include a metadata header `Meili-Include-Metadata` in your search requests. This will add a metadata field to the response, which contains a unique identifier (`uid`) for each query, along with a reminder of the `indexUid` and its primary key.

To use this feature, include the header in your search request:

```http
GET /indexes/{indexUid}/search
Meili-Include-Metadata: true
```

The response will include the metadata:

```json
{
  "hits": [
    {
      "id": 1,
      "title": "Example Document"
    }
  ],
  "metadata": {
    "uid": "unique-query-id",
    "indexUid": "your-index-uid",
    "primaryKey": "your-primary-key"
  },
  "processingTimeMs": 2
}
```

### Multimodal Embeddings
You can now index and search non-textual documents, including images, using multimodal embeddings. This feature allows for AI-powered searches that provide a common semantic representation for images, texts, and other data types.

To enable the `multimodal` experimental feature, use the following command:

```bash
curl \

 -H 'Content-Type: application/json' \
 --data-binary '{
```
