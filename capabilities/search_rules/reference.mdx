Search rules are experimental. Enable them with `PATCH /experimental-features` before using the `/dynamic-search-rules` endpoints.

Use this request body:

```json
{
  "dynamicSearchRules": true
}
```

If the feature is disabled, the `/dynamic-search-rules` routes are not usable and saved rules do not apply at search time.

## Endpoints

All search rules endpoints require an admin API key.

| Method | Route | Purpose |
| --- | --- | --- |
| `POST` | `/dynamic-search-rules` | List rules with pagination and filters |
| `GET` | `/dynamic-search-rules/{uid}` | Get one rule |
| `PATCH` | `/dynamic-search-rules/{uid}` | Create or update a rule |
| `DELETE` | `/dynamic-search-rules/{uid}` | Delete a rule |

## List rules

Send a `POST` request to `/dynamic-search-rules` with pagination and optional filters:

```json
{
  "offset": 0,
  "limit": 20,
  "filter": {
    "attributePatterns": ["promo*"],
    "active": true
  }
}
```

Request fields:

| Field | Type | Description |
| --- | --- | --- |
| `offset` | integer | Number of results to skip |
| `limit` | integer | Maximum number of rules to return |
| `filter.attributePatterns` | array of strings | Filters rule UIDs using attribute-pattern syntax such as `promo*` |
| `filter.active` | boolean | Filters by active or inactive rules |

Response fields:

| Field | Type | Description |
| --- | --- | --- |
| `results` | array | Returned rules |
| `offset` | integer | Applied offset |
| `limit` | integer | Applied limit |
| `total` | integer | Total number of matching rules |

## Get one rule

`GET /dynamic-search-rules/{uid}` returns the stored rule for that UID.

## Create or update a rule

`PATCH /dynamic-search-rules/{uid}` behaves as an upsert:

- It returns `201 Created` when the rule does not exist yet
- It returns `200 OK` when updating an existing rule

Important behavior:

- The route is partial, so omitted fields stay unchanged
- The `uid` comes from the path, not from the request body
- `actions` are required when creating a new rule
- `actions` cannot be `null`
- `actions` cannot be an empty array
- `conditions` are optional, and multiple conditions are combined with `AND`

Example request body:

```json
{
  "description": "Black Friday 2026 rules",
  "priority": 10,
  "active": true,
  "conditions": [
    { "scope": "query", "isEmpty": true },
    { "scope": "time", "start": "2026-11-27T00:00:00Z", "end": "2026-11-27T23:59:59Z" }
  ],
  "actions": [
    {
      "selector": { "indexUid": "products", "id": "123" },
      "action": { "type": "pin", "position": 1 }
    }
  ]
}
```

## Delete a rule

`DELETE /dynamic-search-rules/{uid}` returns `204 No Content` on success.

## Rule object

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `uid` | string | Yes | Unique rule identifier |
| `description` | string | No | Human-readable description |
| `priority` | integer | No | Lower numeric values take precedence over higher ones |
| `active` | boolean | No | Whether the rule is active. Defaults to `true` |
| `conditions` | array | No | Activation conditions. Multiple conditions are combined with `AND` |
| `actions` | array | Yes | Actions to apply when the rule matches |

Search rules are instance-level objects. They are not stored as per-index settings.

## Query condition

Use a query condition when you want a rule to match query text or an empty query:

```json
{ "scope": "query", "contains": "invoice" }
```

or:

```json
{ "scope": "query", "isEmpty": true }
```

Rules:

- A query condition must use either `contains` or `isEmpty`
- A query condition cannot use both
- A query condition cannot omit both

## Time condition

Use a time condition when you want a rule to activate during a specific window:

```json
{
  "scope": "time",
  "start": "2026-06-01T00:00:00Z",
  "end": "2026-06-30T23:59:59Z"
}
```

Rules:

- `start` and `end` must use RFC 3339 timestamps
- `start` and `end` are optional individually
- `start` must not be later than `end`

## Pin action

Search rules currently support one action type: `pin`.

Each action entry contains a selector and an action definition:

```json
{
  "selector": { "indexUid": "products", "id": "summer-sale-landing-page" },
  "action": { "type": "pin", "position": 0 }
}
```

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `selector.id` | string | Yes | Document identifier |
| `selector.indexUid` | string | No | Index to search when you need to target a specific index |
| `action.type` | string | Yes | Action type. The only supported value is `pin` |
| `action.position` | integer | Yes | Target position in the result list. For example, `0` means the first result position |

## Behavioral notes

- Pinned documents can appear even if they are not lexical matches for the full query
- Pinned documents still have to survive the current filters
- A pinned document should not appear twice if it is also returned organically
- Pagination and facet distribution remain coherent with the final result set
