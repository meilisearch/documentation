Meilisearch uses API keys to authenticate requests. Every Meilisearch Cloud project comes with three default API keys that you can use immediately.

## Default API keys

Meilisearch Cloud automatically creates three API keys for each project:

| Key | Purpose | Safe for frontend? |
|-----|---------|-------------------|
| **Default Search API Key** | Search operations only | Yes |
| **Default Admin API Key** | All operations (indexing, settings, etc.) | No |
| **Default Chat API Key** | Chat completions and search | Yes |

Find your API keys in the Cloud dashboard under **Project Settings** > **API Keys**.

## API keys

## Authenticating requests

Add your API key to requests using the `Authorization` header with a Bearer token:

```bash
curl "${MEILISEARCH_URL}/indexes" \
  -H "Authorization: Bearer ${MEILISEARCH_API_KEY}"
```

### Admin API keys

Meilisearch provides two admin API keys for managing your instance:

- The `Default Admin API Key` grants full access to all Meilisearch operations except API key management. Use it to configure index settings, add documents, and perform other administrative tasks.
- The `Default Read-Only Admin API Key` allows read-only access to the whole database. Use it when you need to retrieve information from your Meilisearch instance without being able to modify it.

<Warning>
Do not expose admin API keys on a public frontend.
</Warning>

### Chat API key

The `Default Chat API Key` is designed for frontend usage with [conversational search](/learn/chat/getting_started_with_chat). It has access to both `search` and `chatCompletions` actions, allowing users to both perform searches and interact with the chat completions feature.

## Creating custom API keys

Create custom keys with specific permissions for different use cases:

```bash
curl -X POST "${MEILISEARCH_URL}/keys" \
  -H "Authorization: Bearer ${MEILISEARCH_API_KEY}" \
  -H 'Content-Type: application/json' \
  --data-binary '{
    "name": "Mobile app search key",
    "description": "Search key for mobile application",
    "actions": ["search"],
    "indexes": ["products", "articles"],
    "expiresAt": "2025-12-31T23:59:59Z"
  }'
```

### Key parameters

| Parameter | Description |
|-----------|-------------|
| `name` | Human-readable name for the key |
| `description` | Optional description of the key's purpose |
| `actions` | Array of permitted actions (`search`, `documents.add`, `*`, etc.) |
| `indexes` | Array of indexes the key can access (`*` for all) |
| `expiresAt` | Expiration date (ISO 8601 format) or `null` for no expiration |

For a complete list of available actions, see [API key permissions](/products/platform/api_key_permissions).

## Revoking API keys

Delete a key when it's no longer needed or if it may have been compromised:

```bash
curl -X DELETE "${MEILISEARCH_URL}/keys/KEY_UID" \
  -H "Authorization: Bearer ${MEILISEARCH_API_KEY}"
```

<Tip>
Regularly audit your API keys and remove any that are no longer in use.
</Tip>

## Best practices

### Use the right key for each context

| Context | Recommended key |
|---------|----------------|
| Frontend search | Search API Key |
| Backend indexing | Admin API Key |
| Mobile apps | Custom key with limited scope |
| Third-party integrations | Custom key with specific permissions |

### Security recommendations

- **Never commit API keys to source control** - use environment variables
- **Create scoped keys** for third-party integrations instead of sharing the Admin key
- **Set expiration dates** on keys when possible
- **Use tenant tokens** for multi-tenant applications
